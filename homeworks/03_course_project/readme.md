# Курсовая работа по ETL

Курсовая работа нужна для того, чтобы собрать вместе все, что мы делали в течение курса:

- обычный ETL;
- Airflow;
- PySpark;
- слои `raw / bronze / silver / gold`;
- проверки качества данных;
- инкрементальные запуски;
- streaming / CDC;


## Общая постановка

Нужно сделать небольшой ETL-проект для интернет-магазина.

Есть несколько исходных таблиц:

- `users.csv` - пользователи;
- `products.csv` - товары;
- `orders.csv` - заказы;
- `order_items.csv` - состав заказов;
- `cdc_orders/*.jsonl` - события изменения заказов.

Данные можно сгенерировать готовым скриптом:

```bash
python generate_data.py --variant 1 --output-dir data/raw
```

Номер варианта берется по номеру в списке группы.

## Что нужно сделать

Нужно построить пайплайн, который:

1. генерирует или читает исходные данные;
2. сохраняет сырые данные в `bronze`;
3. чистит данные и сохраняет результат в `silver`;
4. строит витрины в `gold`;
5. обрабатывает CDC-события по заказам;
6. собирает актуальное состояние заказов;
7. строит одну витрину по streaming/CDC-части;
8. выполняет проверки качества;
9. нормально переживает повторный запуск.

Можно делать в ноутбуке, но итоговое решение должно быть оформлено как проект, а не как набор случайных ячеек.

## Слои данных

### Bronze

В `bronze` складываем сырые данные почти как есть.

Нужно сохранить:

- все CSV-файлы;
- все CDC JSONL-файлы;
- технические поля, если они нужны: дата загрузки, имя файла, время загрузки.

### Silver

В `silver` должны лежать уже очищенные данные.

Минимально нужно:

- привести типы;
- убрать дубли заказов;
- убрать строки с `null` в ключевых полях;
- убрать отрицательные количества;
- убрать строки `order_items`, где нет товара;
- убрать заказы, где нет пользователя;
- отдельно собрать актуальное состояние заказа по CDC-событиям.

### Gold

В `gold` лежат витрины.

Нужно сделать:

- минимум 3 обычные batch-витрины;
- минимум 1 витрину на основе CDC/current state;
- одну обязательную витрину из таблицы вариантов.

У каждой витрины в отчете должно быть коротко написано:

- какая у нее гранулярность;
- какие поля в ней есть;
- на какой вопрос она отвечает.

## Генератор данных

В папке лежит файл:

- [`generate_data.py`](./generate_data.py)

Его можно использовать сразу.

Пример:

```bash
python generate_data.py --variant 7 --output-dir data/raw
```

Скрипт создаст:

```text
data/raw/
├── users.csv
├── products.csv
├── orders.csv
├── order_items.csv
├── cdc_orders/
│   ├── batch_001_create.jsonl
│   ├── batch_002_update.jsonl
│   ├── batch_003_update_delete_duplicate.jsonl
│   └── batch_004_late_event.jsonl
├── variant_config.json
├── README_generated.md
└── data_dictionary.md
```

В данных специально есть проблемы:

- дубли заказов;
- `null` в ключевых полях;
- товары без категории;
- строки с несуществующим товаром;
- отрицательные количества;
- отмененные и возвращенные заказы;
- повторное CDC-событие;
- позднее CDC-событие;
- delete-событие.

Их не надо удалять из генератора. Их нужно обработать в ETL.

## Batch-часть

В batch-части обязательно использовать:

- `SparkSession`;
- явные схемы при чтении;
- `DataFrame API`;
- минимум 2 `join`;
- минимум 2 агрегации;
- минимум 1 оконную функцию;
- запись в `Parquet`;
- `partitionBy` хотя бы для одной витрины;
- `explain("formatted")` хотя бы для одного важного запроса.

Нужно коротко объяснить, где возникает `shuffle` и где можно использовать `broadcast join`.

## CDC / streaming-часть

Нужно обработать файлы из `cdc_orders`.

Требуется показать:

- чтение событий;
- дедупликацию по `event_id`;
- обработку `create`, `update`, `delete`;
- сборку актуального состояния заказа;
- обработку late event;
- checkpoint или понятную имитацию микробатчей;
- одну витрину по актуальному состоянию.

Можно делать через `Structured Streaming`. Если с ним сложно, можно сделать учебную версию по микробатчам: читать JSONL-файлы по очереди и после каждого батча обновлять состояние. Главное - явно показать логику.

## Airflow

Нужно добавить DAG или упрощенный локальный сценарий запуска.

Минимальный DAG:

1. сгенерировать данные;
2. загрузить bronze;
3. построить silver;
4. построить gold;
5. обработать CDC;
6. запустить проверки качества.

Если Airflow не запускается локально, можно приложить код DAG и описать, как он должен работать.

## Проверки качества

Нужно реализовать проверки в коде.

Минимальный набор:

- итоговые витрины не пустые;
- нет `null` в ключевых полях;
- нет отрицательной выручки;
- нет отрицательных количеств после очистки;
- нет дублей там, где должен быть уникальный ключ;
- orphan-строки не попали в итоговые расчеты;
- повторный запуск не удваивает данные;
- позднее событие не портит актуальное состояние заказа;
- delete-событие учитывается.
