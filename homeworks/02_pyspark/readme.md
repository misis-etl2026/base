# Домашнее задание №2: PySpark ETL

Необходимо реализовать ETL-пайплайн на `PySpark` для данных интернет-магазина.

Цель задания:
- научиться читать и обрабатывать данные в `PySpark`;
- закрепить работу с очисткой данных, `join`, агрегациями и записью в `Parquet`;
- показать понимание того, как `PySpark` используется в типичном ETL-сценарии;
- научиться оформлять решение как воспроизводимый и понятный проект, а не как набор разрозненных ячеек.

## Постановка задачи

В рамках задания вам дан генератор сырых данных:

- [`generate_data.py`](/Users/avmysh/projs/2026/etl/homeworks/02_pyspark/generate_data.py)

Он создаёт 4 CSV-файла:
- `users.csv`
- `products.csv`
- `orders.csv`
- `order_items.csv`

Сгенерированные данные намеренно содержат ошибки и неоднозначности, характерные для реальных источников:
- дубликаты заказов;
- `null` в ключевых полях;
- отрицательные количества;
- строки без соответствующего пользователя или товара;
- товары без категории;
- отменённые заказы.

## Ожидаемая схема таблиц

### 1. `users.csv`

Описание:
- справочник пользователей

Схема:

| Поле | Тип | Описание |
|---|---|---|
| `user_id` | `int` | идентификатор пользователя |
| `country` | `string` | страна пользователя |
| `signup_date` | `date` | дата регистрации |

### 2. `products.csv`

Описание:
- справочник товаров

Схема:

| Поле | Тип | Описание |
|---|---|---|
| `product_id` | `int` | идентификатор товара |
| `category` | `string` | категория товара |
| `price` | `double` | цена товара |

Примечание:
- в части строк `category` может быть `null`.

### 3. `orders.csv`

Описание:
- заказы интернет-магазина

Схема:

| Поле | Тип | Описание |
|---|---|---|
| `order_id` | `string` | идентификатор заказа |
| `user_id` | `int` | идентификатор пользователя |
| `order_ts` | `timestamp` | дата и время заказа |
| `status` | `string` | статус заказа |

Возможные статусы:
- `completed`
- `pending`
- `shipped`
- `cancelled`

Примечание:
- в таблице могут быть дубликаты по `order_id`;
- в части строк `user_id` может быть `null`.

### 4. `order_items.csv`

Описание:
- состав заказа

Схема:

| Поле | Тип | Описание |
|---|---|---|
| `order_id` | `string` | идентификатор заказа |
| `product_id` | `int` | идентификатор товара |
| `quantity` | `int` | количество товара в заказе |

Примечание:
- часть строк может содержать `null` в `product_id`;
- часть строк может содержать отрицательное или нулевое `quantity`;
- часть строк может ссылаться на несуществующий `product_id`.

## Требуется реализовать

Необходимо написать `PySpark`-приложение, которое:

1. читает все 4 исходные таблицы;
2. использует явную `schema` при чтении данных;
3. приводит типы данных;
4. очищает данные;
5. формирует очищенный слой данных;
6. строит 2 аналитические витрины и 1 пользовательскую витрину;
7. сохраняет результат в `Parquet`;
8. показывает базовый performance-разбор результата.

## Обязательные требования к очистке данных

В решении обязательно должны быть реализованы следующие шаги:

1. удалить дубликаты заказов по `order_id`;
2. удалить или отфильтровать строки с `null` в ключевых полях;
3. удалить строки с `quantity <= 0`;
4. исключить отменённые заказы, где `status = 'cancelled'`;
5. привести `order_ts` к типу `timestamp`;
6. выделить поле с датой заказа;
7. исключить строки `order_items`, для которых нет товара в `products`;
8. исключить строки `orders`, для которых нет пользователя в `users`.

## Обязательные аналитические витрины

В результате обработки необходимо построить 3 витрины.

### Витрина 1: `daily_sales_mart`

Гранулярность:
- день + страна

Поля:

| Поле | Тип | Описание |
|---|---|---|
| `date` | `date` | дата |
| `country` | `string` | страна |
| `orders_cnt` | `long` | количество заказов |
| `buyers_cnt` | `long` | количество уникальных покупателей |
| `items_cnt` | `long` | количество купленных единиц товара |
| `revenue` | `double` | выручка |

### Витрина 2: `category_sales_mart`

Гранулярность:
- день + категория

Поля:

| Поле | Тип | Описание |
|---|---|---|
| `date` | `date` | дата |
| `category` | `string` | категория |
| `orders_cnt` | `long` | количество заказов |
| `items_cnt` | `long` | количество проданных единиц |
| `revenue` | `double` | выручка |

### Витрина 3: `user_sales_mart`

Гранулярность:
- пользователь

Поля:

| Поле | Тип | Описание |
|---|---|---|
| `user_id` | `int` | пользователь |
| `country` | `string` | страна |
| `orders_cnt` | `long` | число заказов |
| `revenue` | `double` | выручка по пользователю |
| `first_order_ts` | `timestamp` | первый заказ |
| `last_order_ts` | `timestamp` | последний заказ |

## Обязательные технологические требования

В решении обязательно должны присутствовать:
- `SparkSession`;
- чтение CSV в `PySpark` с явным описанием схемы;
- `DataFrame API`;
- `join`;
- `withColumn`;
- `groupBy` и `agg`;
- запись результата в `Parquet`;
- запись результатов с `partitionBy("date")` для витрин, где это применимо;
- `explain("formatted")` хотя бы для одной ключевой витрины;
- использование `broadcast join` с кратким объяснением;
- реализация хотя бы одной витрины через `Spark SQL`.

## Обязательные проверки качества данных

В решении должны быть явно реализованы и показаны следующие проверки:
- после очистки `order_id` уникален;
- в итоговых витринах нет `null` в ключевых полях;
- `revenue >= 0`;
- итоговые витрины не пустые;
- строки `order_items` без товара не попадают в итоговые расчёты;
- строки `orders` без пользователя не попадают в итоговые расчёты.

Проверки могут быть реализованы через:
- `assert`;
- логирование;
- отдельный блок в коде перед записью результата.
